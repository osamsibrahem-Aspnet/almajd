# Deploying Almajd to MonsterASP.NET (Free Tier)

This is a one-time setup guide. After this, redeploys are just "rebuild + re-upload".

## What you're deploying

- **.NET 8 API** → MonsterASP IIS site, served at `/api/*`
- **Angular 17 frontend** → same IIS site, served from root (`/`)
- **SQL Server database** → MonsterASP MSSQL instance
- **Seed data** → applied automatically on first boot (`DbInitializer.SeedingDataAsync`)

## Step 1 — Create the MonsterASP site + database

1. Log into [monsterasp.net](https://www.monsterasp.net) with `osamsibrahem@gmail.com`.
2. **Create a new site** under your free plan. Note down the URL it gives you, e.g. `https://almajd-001-site1.gtempurl.com` (the exact format varies — it might end in `.monsterasp.net` or `.gtempurl.com`).
3. **Create a new MSSQL database**. The control panel will show you a connection string like:
   ```
   Data Source=db12345.public.databaseasp.net; Initial Catalog=db12345; User Id=db12345; Password=Abc-XYZ-123; Encrypt=False; TrustServerCertificate=True;
   ```
4. Note down the **FTP host / username / password** for the site — you'll need it to upload files.

## Step 2 — Fill in the three placeholders

Edit `Almajd.Api/appsettings.Production.json` and replace:

| Placeholder | What to put |
|---|---|
| `REPLACE_WITH_MONSTERASP_CONNECTION_STRING` | The exact connection string from Step 1.3 |
| `https://REPLACE_WITH_YOUR_MONSTERASP_SITE` (3 places) | Your site URL from Step 1.2, **without** trailing slash for CORS, **with** trailing slash for `ApiUrl`/`ClientUrl` |
| `DefaultAdmin.Password` | Change `ChangeMe_StrongPassword_2026!` to something only you know |

**JWT key and OTP hash key** are already generated and baked in — don't touch them. They must NOT change between deploys or all existing JWTs and pending OTPs become invalid.

Edit `almajd-frontend/src/environments/environment.production.ts`:
```typescript
export const environment = {
  production: true,
  apiUrl: 'https://YOUR-MONSTERASP-SITE'  // same root URL as above, no trailing slash
};
```

> **Important**: since the API and Angular share one domain, `apiUrl` here is the **full origin** (e.g. `https://almajd.monsterasp.net`). The Angular service code already prefixes `/api/...` paths.

## Step 3 — Build the backend

From `Almajd/`:
```bash
dotnet publish Almajd.Api/Almajd.Api.csproj -c Release -o ./publish-api
```

Output goes to `./publish-api/`. This folder is what you upload.

## Step 4 — Build the frontend

From `Almajd/almajd-frontend/`:
```bash
npx ng build --configuration production
```

Output goes to `./dist/almajd-frontend/`. Note the inner folder structure — there's usually a `browser/` subfolder. Confirm where `index.html` lands; that's the directory you upload.

## Step 5 — Combine into one IIS site layout

MonsterASP serves your site from `wwwroot/`. Lay out the publish folder like this:

```
publish-final/
├── Almajd.Api.dll
├── Almajd.Application.dll
├── Almajd.Infrastructure.dll
├── Almajd.Domain.dll
├── appsettings.json
├── appsettings.Production.json
├── web.config              ← auto-generated by `dotnet publish`
├── ... (other DLLs)
└── wwwroot/
    ├── index.html          ← from Angular build
    ├── main.[hash].js
    ├── styles.[hash].css
    ├── assets/
    │   └── i18n/
    │       ├── en.json
    │       └── ar.json
    ├── BrandLogos/         ← runtime-created (empty on first deploy is fine)
    ├── InvoicePdfs/
    ├── KycDocuments/
    └── ProductImages/
```

Steps:
1. Copy everything from `publish-api/` into `publish-final/`.
2. Copy everything from `almajd-frontend/dist/almajd-frontend/browser/*` into `publish-final/wwwroot/` (merge with the existing `wwwroot/` that the API publish creates).
3. Make sure `index.html` is at `publish-final/wwwroot/index.html`.

## Step 6 — Fix `web.config` for SPA fallback

MonsterASP needs the IIS site to:
- Serve `/api/*` to ASP.NET
- Serve everything else from `wwwroot/` (with SPA fallback to `index.html` for client-side routes)

The `web.config` that `dotnet publish` generates handles `/api/*` automatically. For the SPA fallback, edit `web.config` to add a URL Rewrite rule before the ASP.NET handler:

```xml
<configuration>
  <system.webServer>
    <handlers>
      <add name="aspNetCore" path="*" verb="*" modules="AspNetCoreModuleV2" resourceType="Unspecified" />
    </handlers>
    <aspNetCore processPath="dotnet" arguments=".\Almajd.Api.dll" stdoutLogEnabled="false" hostingModel="inprocess" />

    <rewrite>
      <rules>
        <!-- Don't rewrite API calls -->
        <rule name="API" stopProcessing="true">
          <match url="^api/.*" />
          <action type="None" />
        </rule>
        <!-- Don't rewrite static asset requests with a file extension -->
        <rule name="StaticAssets" stopProcessing="true">
          <match url=".*\..+$" />
          <action type="None" />
        </rule>
        <!-- Everything else → Angular SPA -->
        <rule name="AngularRoutes" stopProcessing="true">
          <match url=".*" />
          <action type="Rewrite" url="/index.html" />
        </rule>
      </rules>
    </rewrite>
  </system.webServer>
</configuration>
```

> Keep the existing `<aspNetCore>` line that `dotnet publish` generated — don't replace it; just add the `<rewrite>` block.

## Step 7 — Upload via FTP

Using FileZilla / WinSCP / any FTP client:
1. Connect with the MonsterASP FTP credentials (host/user/pass from Step 1.4).
2. Upload the **contents** of `publish-final/` (not the folder itself) into the site's root directory (usually `httpdocs/` or `wwwroot/` on their end — check their docs).

## Step 8 — First boot

Visit `https://YOUR-MONSTERASP-SITE/` in a browser.

What happens automatically:
1. IIS starts the .NET process.
2. `Program.cs` reads `appsettings.Production.json`.
3. `DbInitializer.SeedingDataAsync()` runs:
   - Applies all EF migrations (creates all tables).
   - Creates the 8 ASP.NET Identity roles.
   - Creates the admin user from `DefaultAdmin` config.
   - Creates 7 staff test users (one per non-Admin role): `salesrep@almajd.test`, `warehouseop@almajd.test`, etc. — all with password `Test@12345`.
   - Seeds MAIN warehouse + 4 locations.
   - Seeds default price list (EGP).
   - Seeds invoice counter for current year.
   - Seeds 7 notification templates.
   - Seeds 3 brands, 7 categories, 5 products, 7 SKUs, 2 suppliers + supplier-SKU links, 3 customers, stock items for every SKU.
4. Angular `index.html` is served from `wwwroot/`.

**This may take 20–60 seconds on the very first request** while migrations run. Subsequent requests are fast.

## Troubleshooting

**500 on every request** — check MonsterASP's IIS log (control panel → Logs). Most common: connection string typo.

**Login page works but every API call returns CORS error** — your site URL in `Cors:AllowedOrigins` doesn't match exactly. Must be the exact origin (scheme + host, no trailing slash, no path).

**`IDX10720: ... 208 bits`** — the JWT key got truncated during the JSON edit. Re-paste the full base64 value.

**Site shows "503 Service Unavailable" after migration** — the seeder probably crashed. Check the IIS log. Often it's because the MonsterASP MSSQL plan doesn't allow `CREATE LOGIN` style operations — the `DbInitializer` only creates app data (not server-level objects), so this should be fine, but check.

**Frontend works but `/admin/dashboard` shows 404 on reload** — the SPA fallback in `web.config` isn't matching. Re-check Step 6.

**Database connection string failing** — MonsterASP's free DBs sometimes require `Encrypt=False;TrustServerCertificate=True` in the connection string. If it's not in the string they provide, append it.

## Redeploy procedure

After the first deploy, future updates are:
1. `dotnet publish Almajd.Api/Almajd.Api.csproj -c Release -o ./publish-api`
2. `cd almajd-frontend && npx ng build --configuration production`
3. Replace the DLLs in the FTP root.
4. Replace `wwwroot/` contents with the new Angular build (don't delete `BrandLogos/` and other upload subfolders!).
5. Visit any page once to wake IIS up.

EF migrations apply automatically on boot, so schema changes don't need any extra step.

## Security note

`appsettings.Production.json` now contains your JWT key and OTP pepper. **Do not commit this file to a public Git repository**. The `.gitignore` should exclude `appsettings.Production.json` — verify before pushing.
